## <pattern_registry>

האינדקס של התבניות מוחזק בקובץ **מובנה** (`registry.yaml`), לא בפרוזה מפוזרת, כדי
שבחירת "התבנית בעלת הציון הגבוה ביותר שמתאימה למשימה" תהיה query ולא קריאה ידנית של
קובץ אחרי קובץ (ראה [`connector-policy.md`](./connector-policy.md) ›
`<information_sources>`).

כל רשומה כוללת:

```yaml
- id: stripe-subscription-webhook
  domain: stripe            # api / auth / stripe / database / ui / ai-agents …
  status: active            # candidate | active | deprecated | retired
  score: 87                 # 0–100, נגזר מהקריטריונים למטה
  version: 2.1.0
  used_in: 4                # מספר פרויקטים/הקשרים אמיתיים שבהם שימשה
  evidence:                 # תוצאות אמת — לא הערכה
    - "prod 6 חודשים, 0 incidents — project X"
    - "החליפה מימוש ידני שגרם ל-double-charge — לקח LL-014"
  code: ../patterns/stripe/subscription-webhook/
  test: ../patterns/stripe/subscription-webhook/test/
  source: ../lessons-learned/bugs/LL-014.md   # לקח/ADR שממנו עלתה, אם רלוונטי
```

`evidence` ו-`used_in` הם מה שמבדיל תבנית מדורגת מ-snippet אקראי: הציון נשען על
תוצאות אמת, באותו מודל-ראיה של רמות הביטחון ב-[`learning-loop.md`](./learning-loop.md).

</pattern_registry>

## <scoring>

הציון נגזר משישה קריטריונים, כל אחד מבוסס על תוצאות אמת מ-`evidence` — לא על הערכה
תיאורטית:

- **Reliability** — האם החזיקה בפרודקשן בלי incidents.
- **Maintainability** — כמה קל היה להתאים/לתחזק אותה בפרויקטים שונים.
- **Security** — עברה סריקה/סקירת אבטחה; אין דליפות או חורים ידועים.
- **Performance** — עומדת בדרישות ביצועים בשימוש אמיתי.
- **Production success rate** — יחס השימושים שהצליחו מול אלה שדרשו תיקון/חזרה.
- **Reusability** — שימשה ביותר מהקשר אחד בלי שכתוב מהותי.

### איך הציון מתעדכן

הציון אינו נקבע פעם אחת — הוא **תוצר של תוצאות**. בכל פרויקט שמשתמש בתבנית, רשום את
התוצאה ב-`evidence` ועדכן את הציון:

- **תוצאה חיובית** (עבדה בפרודקשן, התאמה קלה) → העלה ציון, הגדל `used_in`.
- **נדרשה התאמה כבדה / כמעט-שכתוב** → סימן ל-Maintainability/Reusability נמוכים → הורד.
- **גרמה ל-incident / כשל אבטחה** → הורד משמעותית; שקול `deprecated` (ראה
  `<lifecycle>`), ופתח לקח ב-[`learning-loop.md`](./learning-loop.md).

תבנית בעלת ציון גבוה מועדפת על נמוכה (ראה היררכיית השימוש ב-README ›
*Usage Priority Rules*: patterns › external systems › reference repos › templates ›
new implementation).

</scoring>

## <lifecycle>

תבנית עוברת ארבעה מצבים. לכל מעבר יש קריטריון מפורש — לא החלטה לפי תחושה.

```
Candidate  ──proven──▶  Active  ──superseded / failures──▶  Deprecated  ──no users──▶  Retired
```

### Candidate (מועמדת)

**איך נכנסת — רק כשאין תבנית מתאימה קיימת.** לפני הוספה, חפש ב-registry; אם קיימת
תבנית מתאימה — השתמש/הרחב אותה במקום ליצור חדשה. תבנית חדשה נכנסת רק כש:

- אין תבנית קיימת שעונה על הצורך, **או**
- היא עלתה כ-Best Practice מלקח מאומת (ראה [`learning-loop.md`](./learning-loop.md) ›
  הבשלת ידע).

**דרישות כניסה:** קוד עובד + טסט שמכסה אותו + ראיה מ-≥1 שימוש אמיתי. עדיין **לא**
מועדפת אוטומטית — היא בהמתנה להוכחה חוצת-הקשרים.

### Active (פעילה)

**מעבר מ-Candidate:** הוכחה ב-≥2 הקשרים/פרויקטים אמיתיים עם תוצאה חיובית, וקיבלה
ציון. זו התבנית שנשלפת לשימוש-חוזר כברירת מחדל.

**עדכון בזמן Active:** כל שינוי בקוד התבנית = **version bump** + הרצת הטסט מחדש (חייב
לעבור) + הערה ב-`evidence`. אל תערוך בשקט תבנית שאחרים נשענים עליה בלי re-validation.

### Deprecated (לא מומלצת לחדש)

**מתי:** הציון צנח מתחת לסף, נמצא בה חור אבטחה/נכונות, או הגיעה מחליפה טובה יותר
ל-Active. **משמעות:** אל תשתמש בה לעבודה חדשה; שימושים קיימים מהגרים בהדרגה. רשום
ב-registry מי המחליפה ולמה.

### Retired (הוסרה מהבחירה)

**מתי:** אין שימושים פעילים שנותרו, או שעברה תקופה כ-Deprecated והוחלפה במלואה.
מוסרת מהבחירה ב-registry (`status: retired`). **הקוד נשאר בהיסטוריית git** — אפשר
לשחזר אם צריך. אין מחיקה אמיתית.

### תקציר קריטריוני המעבר

| מעבר | תנאי |
|---|---|
| → Candidate | אין תבנית מתאימה / עלתה מ-Best Practice; יש קוד+טסט+ראיה אחת |
| → Active | הוכחה ב-≥2 הקשרים עם תוצאה חיובית; קיבלה ציון |
| → Deprecated | ציון מתחת לסף / כשל אבטחה-נכונות / הגיעה מחליפה ב-Active |
| → Retired | אין שימושים פעילים / הוחלפה במלואה (נשמרת ב-git) |

</lifecycle>

## חיבור ללולאת הלמידה

מחזור החיים והדירוג כאן הם הצד ה"חיובי" של אותה מערכת ש-
[`learning-loop.md`](./learning-loop.md) מנהל בצד ה"שלילי": לקח מאומת ברמת Best
Practice **נכנס כתבנית Candidate** עם הראיה שכבר נאספה; וכישלון של תבנית בפרודקשן
**פותח לקח** ומוריד את הציון. אותו מודל-ראיה (תוצאות אמת + מספר ההקשרים) מזין את
שניהם — כך הידע שנצבר הופך גם לקוד לשימוש-חוזר וגם לאזהרות ממה להימנע.
