## <information_sources>

מאיפה מושכים מידע ודוגמאות — לפי סדר העדיפות:

1. **ריפו ה-Engineering OS** — [`../patterns/`](../patterns/) ו-
   [`../templates/`](../templates/): תבניות קוד וארכיטקטורות מאושרות שכבר אומתו
   אצלנו. בדוק כאן לפני כל כתיבה מאפס. ראה גם [`../docs/`](../docs/) למאגר התיעוד
   המאושר.
2. **Context7** — תיעוד רשמי עדכני ודוגמאות קוד של ספריות ופריימוורקים.
3. **קונקטור ייעודי למשימה** — אם קיים קונקטור שמתמחה בנושא (ראה רשימת הקונקטורים למטה).
4. **חיפוש רשת** — רק כשהמקורות לעיל אינם מספיקים.

הסיבה לסדר: `patterns/`/`templates/` הם קוד שכבר אומת אצלנו; Context7 הוא תיעוד
נוכחי ומדויק; ידע פנימי בלבד עלול להיות מיושן.

</information_sources>

## <connectors>

הקונקטורים הם **מקור האמת** לבדיקת מצב המערכת. **בחירת הקונקטור נעשית לפי המשימה
שהמשתמש ביקש.** שני סוגים: **ליבה קבועה** (כמעט בכל פרויקט) ו**תלויי-פרויקט**
(מפעילים לפי סוג המוצר והמשימה).

**נוהל קונקטור לא זמין (fallback):**

כשהקונקטור הדרוש אינו מחובר, עבור על השלבים לפי הסדר ועצור ברגע שנמצא פתרון:

1. **חלופה מחוברת** — חפש קונקטור אחר שמבצע את אותן פעולות (חלופה מאותה קטגוריה).
   אם הוא מחובר — השתמש בו ועדכן את המשתמש שעברת אליו.
2. **Composio** — אם אין חלופה מחוברת, בדוק אם החיבור קיים דרך **Composio** (שמספק
   שרתי MCP רבים). אם כן — השתמש בו דרך Composio.
3. **פנייה למשתמש** — רק אם אין חלופה מחוברת וגם Composio לא מספק את החיבור, עצור
   ועדכן את המשתמש שהקונקטור דרוש למשימה ואינו זמין.
4. אם המשתמש משיב שאין לו דרך לחבר כרגע — חזור על שלבים 1–2 עבור החלופה הבאה, עד
   שתמצא פתרון מחובר או שייגמרו החלופות. אם אין אף פתרון — עצור ועדכן שאי אפשר לבצע
   את המשימה בלי חיבור.

מקרא סטטוס: ✅ מחובר · ⏳ להתחבר · 🔍 לבחון

### ליבה קבועה

**GitHub** — ✅ מחובר
- תפקיד: ניהול הריפו — קוד, branches, PRs, קומיטים, היסטוריה.
- מתי: בכל אינטראקציה עם הריפו.
- מדיניות branch/merge מלאה: ראה [`git-policy.md`](./git-policy.md).

**Context7** — ✅ מחובר
- תפקיד: משיכת תיעוד רשמי עדכני ודוגמאות קוד של ספריות ופריימוורקים.
- מתי: בתחילת פרויקט, ובכל פעם שעובדים עם ספרייה/גרסה לא מוכרת.
- הסיבה: מונע שימוש ב-API ישנים מתוך ידע מיושן — מושך את התיעוד הנוכחי בפועל.

**Sentry** — ✅ מחובר
- תפקיד: ניטור ואיתור שגיאות/קריסות בזמן אמת בפרודקשן (stack traces, נתוני רקע).
- מתי: **כל דיבאג מתבצע דרך Sentry (או קונקטור דיבאג ייעודי) — לא על ידי קלוד לבד.**
  ראה [`debugging-policy.md`](./debugging-policy.md) › `<debug_loop>`.

### תלויי-פרויקט

הפעל לפי סוג המוצר והמשימה. הקטגוריות שלהלן אינן ממצות — לסוג פרויקט או צורך שאינו
מכוסה (למשל ML, אוטומציה), בחר את הקונקטור שמתאים למשימה לפי אותו היגיון, והפעל את
נוהל ה-fallback אם אינו מחובר.

#### Backend / Data / אחסון

**Supabase** — ✅ מחובר
- תפקיד: בסיס הנתונים של הפרויקט; בדיקת data אמיתי ואימות state.
- מתי: בכל בדיקת מצב נתונים — אל תנחש, בצע query.
- הערה: עלות תלויה בסקייל של האפליקציה (תוכנית חינמית קיימת).

**Firebase** — ⏳ להתחבר
- תפקיד: אימות משתמשים, Analytics, Crashlytics, Cloud Messaging (FCM),
  Remote Config, A/B Testing, App Distribution.
- מתי: פרויקטי mobile/web שצריכים auth מנוהל, push, או ניטור קריסות מובנה.

**Pinecone** — ⏳ להתחבר
- תפקיד: vector DB לחיפוש סמנטי ולעבודה עם embeddings.
- מתי: יישומי AI עם RAG, חיפוש סמנטי, או שילוב מודלי שפה.

**Upstash** — ⏳ להתחבר
- תפקיד: Redis serverless (ללא ניהול שרתים) — מסד נתונים מבוסס-זיכרון (RAM) מהיר.
- מתי: caching של נתונים שנשלפים בתדירות גבוהה כדי להאיץ זמני תגובה, או ניהול תורים
  של משימות רקע (background jobs).

**Clerk** — ⏳ להתחבר
- תפקיד: ניהול הרשמה ואימות משתמשים — התחברות, שחזור סיסמה, Social Login (גוגל וכו'),
  עם רכיבי קוד מוכנים להטמעה.
- מתי: כשצריך auth מאובטח בלי לכתוב מאפס. חלופה ל-auth של Firebase/Supabase לפי הפרויקט.

**Prisma** — ⏳ להתחבר
- תפקיד: ORM לעבודה מול בסיס הנתונים — schema, מיגרציות, queries type-safe.
- מתי: גישה ל-DB דרך שכבת ORM במקום SQL גולמי.
- הערה: צריך לחבר ממחשב.

#### פריסה / אירוח

**Vercel** — ✅ מחובר
- תפקיד: deploy של פרונט ובקאנד לענן, בדיקת לוגים ו-deployments.
- מתי: פריסת web (Next.js וכו'), בדיקת מצב deployment.
- הערה: מחובר דרך חשבון GitHub. עדכון משתני סביבה ב-Vercel דרך קלוד עדיין דורש
  הבהרה — בינתיים אם צריך לעדכן משתנה, ציין זאת למשתמש.

**Cloudflare** — ⏳ להתחבר
- תפקיד: DNS, אבטחה (הגנת DDoS), והאצת ביצועים (CDN); Workers / edge.
- מתי: ניהול רשומות רשת, הגנה, או הרצת לוגיקה ב-edge.

**Expo** — ✅ מחובר
- תפקיד: build והעלאת אפליקציות mobile לחנויות Android ו-Apple (React Native).
- מתי: פרויקטי mobile — בנייה ופריסה.
- הערה: תוכנית חינמית מספיקה.

**Azure** — ⏳ להתחבר
- תפקיד: ענן של מיקרוסופט — אירוח, שירותי ענן ותשתית.
- מתי: פרויקטים שדורשים תשתית ענן של מיקרוסופט.
- הערה: יש תקציב חינמי (טוב לפרויקטים אישיים); החיבור דרך המשתמש של אבא + שרת MCP.

**Google Cloud** — 🔍 לבחון
- תפקיד: ענן של גוגל עם יכולת סקיילינג.
- מתי: פרויקטים שדורשים תשתית ענן של גוגל וסקייל גבוה.

#### בדיקות UI

**Playwright** — ⏳ להתחבר
- תפקיד: אוטומציית דפדפן ובדיקות end-to-end (E2E) — לחיצות, טפסים, ניווט.
- מתי: בדיקת flows ב-web (login, checkout). אין פיצ'ר UI ב-web בלי בדיקה.

**Maestro** — ⏳ להתחבר
- תפקיד: אוטומציית בדיקות UI לאפליקציות mobile (iOS ו-Android).
- מתי: המקבילה ל-Playwright עבור mobile.

**Storybook** — ⏳ להתחבר
- תפקיד: פיתוח ובדיקת רכיבי UI בבידוד, וכל המצבים הוויזואליים שלהם.
- מתי: בנייה ובדיקה של קומפוננטות לפני שילובן במוצר המלא.

**Chromatic** — ⏳ להתחבר
- תפקיד: בדיקות ויזואליות (visual regression) בשילוב עם Storybook — מצלם רכיבים,
  משווה לגרסה קודמת, ומתריע על שינויי פיקסלים לא מכוונים.
- מתי: לתפיסת באגים ויזואליים אחרי שינויי UI.

#### בדיקות API

**Postman** — ✅ מחובר
- תפקיד: בדיקת API מקצה-לקצה ושליחת בקשות HTTP.
- מתי: אימות endpoints, בדיקת בקשות/תגובות, ובדיקת אינטגרציות API.

#### אנליטיקה / ניטור AI

**PostHog** — ⏳ להתחבר
- תפקיד: ניתוח התנהגות משתמשים בתוך המוצר (איפה לוחצים, צווארי בקבוק), A/B testing,
  feature flags.
- מתי: הבנת שימוש אמיתי ושיפור חוויית משתמש מבוססת-נתונים.

**Arize** — ⏳ להתחבר
- תפקיד: ניטור מודלי AI בפרודקשן ומעקב ביצועים אחרי שעלו לאוויר (MLOps).
- מתי: פרויקטי AI בפרודקשן שצריך לנטר את ביצועי המודל שלהם.

**Braintrust** — 🔍 לבחון
- תפקיד: הערכה (eval) של אפליקציות LLM — לוודא דיוק ולצמצם הזיות לפני שמגיע למשתמשים.
- מתי: פיתוח ובדיקת איכות של פיצ'רים מבוססי-LLM.

#### ניהול פרויקטים ותשתית

**Notion** — ✅ מחובר
- תפקיד: אפיון ותכנון פרויקטים, תיעוד החלטות, מעקב משימות.
- מתי: **השלב הראשון בכל משימה** — אפיון ותכנון לפני כתיבת קוד.

**Composio** — ✅ מחובר (CLI)
- תפקיד: מספק שרתי MCP רבים שניתן לחבר לקלוד — מרחיב את מאגר הקונקטורים הזמין.
- מתי: כשצריך חיבור לשירות שאין לו קונקטור ישיר — בדוק אם Composio מספק שרת MCP מתאים.

## <environment>

### מקור אמת יחיד למשתנים

**כל** המשתנים והסודות של הפרויקט — בלי יוצא מן הכלל — מרוכזים במקום מאובטח אחד
שממנו ניתן לשלוף אותם כשצריך לעדכן פלטפורמה. אסור לפזר משתנים בין מקומות שונים.

- מקור האמת: קובץ `.env` מקומי (לא נכנס ל-git; מוגן ב-`.gitignore`).
- בריפו נשמר רק `.env.example` עם **שמות** המשתנים וערכי דמה — בלי ערכים אמיתיים.
- אל תכתוב secrets בקוד, בלוגים או בקומיטים. אם נחשף secret — סמן זאת למשתמש מיד.
- הסיבה למקור יחיד: כשמשתנה מתעדכן, מעדכנים במקום אחד ומסנכרנים החוצה — בלי לרדוף
  אחרי עותקים מפוזרים שיוצרים אי-התאמות ובאגים.

### נוהל עדכון משתנה בפלטפורמה

כשצריך להזריק/לעדכן משתנה בפלטפורמה חיצונית (GitHub, Vercel, Supabase וכו'),
פעל לפי סדר העדיפות:

1. **קלוד מעדכן בעצמו** — אם יש קונקטור/הרשאה שמאפשרים זאת, בצע את העדכון ישירות,
   ושלוף את הערך ממקור האמת.
2. **השלמת חיבור** — אם חסר טוקן/חיבור כדי שתוכל לעדכן בעצמך, בקש מהמשתמש להשלים
   אותו (למשל "צור טוקן עם הרשאת X ב-<קישור>"), ואז בצע את העדכון.
3. **המשתמש מעדכן ידנית** — רק אם אי אפשר אחרת. במקרה זה ספק הוראות מדויקות:
   - **קישור מדויק** לדף ההגדרות הספציפי בפלטפורמה.
   - **שם המשתנה** ו**הערך** המדויקים לעדכון (שלוף את הערך ממקור האמת).
   - צעדים ברורים מה ללחוץ והיכן.

</environment>
