## <cleanup>

מחיקת קוד מיותר היא חלק מהעבודה — קוד מת מסתיר באגים ומקשה על תחזוקה. בצע מעבר
ניקוי קבוע לפני כל commit.

### מה למחוק

- קוד שאינו בשימוש: פונקציות, משתנים, imports ו-branches מתים.
- `debug` / `console.log` / prints זמניים שנוספו לצורך פיתוח.
- קוד שהושאר "אולי נצטרך" (dead code) — git שומר את ההיסטוריה, אפשר לשחזר.
- לוגיקה כפולה — אחֵד למקום אחד.

### מה לא למחוק

- קוד שאתה לא בטוח שאינו בשימוש — קודם **ודא** (חיפוש references בפרויקט), אל תנחש.
- קוד שאינו קשור למשימה הנוכחית — אל תנקה דברים שלא נגעת בהם "אגב".
- טסטים — אסור למחוק או לרכך טסט כדי לעקוף כשל; זה מסתיר פונקציונליות חסרה.

### אחרי המחיקה — אמת

מחיקה היא שינוי שעלול לשבור. אחרי ניקוי, הרץ את הבדיקות הרלוונטיות (ראה
`<pre_commit_review>`) כדי לוודא שלא נשבר דבר. מחיקה לא מאומתת אסורה.

אם לא בוצע ניקוי מאומת — הקומיט אינו תקף.

> **אכיפה דטרמיניסטית** (`scripts/enforcement/enforce-quality.sh`, נקרא מ-`pre-commit`):
> ה-diff ה-staged נחסם אם נוספו שאריות debug חד-משמעיות — `debugger`, `breakpoint()`,
> `pdb.set_trace()`/`import pdb`, `binding.pry`/`byebug` — או סמני merge-conflict
> (`<<<<<<<`/`>>>>>>>`). `console.log`/`print` מקבלים **אזהרה** (לא חסימה). שאר ה-cleanup
> (קוד מת, לוגיקה כפולה) שיפוטי ונשאר ידני. bypass: `EOS_BYPASS_CLEANUP=1`.

</cleanup>

## <pre_commit_review>

**לפני כל קומיט, אמת את הקוד בבדיקה — אל תסתמך על קריאה בלבד.** הבדיקה היא שמספקת
את ההוכחה ל"מה עובד / מה לא עובד" שנכנסת לתיעוד הקומיט (ראה
[`git-policy.md`](./git-policy.md) › `<commit_protocol>`).

> **Cadence:** הרץ quality gates **לפני כל commit** — לא לפני כל שינוי בודד ולא רק
> בסוף משימה. אם אתה בולק שינויים קטנים ורצים את הבדיקות כל שעה-שעתיים (יחד עם
> הcommit) — זה הקצב הנכון. הרצה אחת שמכסה diff שלם עדיפה על הרצות תכופות על חצאי-שינויים.

### מה נחשב אימות אמין

- **תוצאה אובייקטיבית מכלי**, לא התרשמות: פלט טסט שעבר, query שהחזיר את הנתון
  הצפוי, לוג נקי, צילום UI שאומת. "הקוד נראה נכון" אינו אימות.
- **בדיקה על קלט מייצג**, לא רק על ה-happy path: כלול לפחות מקרה קצה אחד או קלט לא תקין.
- **בדיקה שמכוונת לשינוי שביצעת** — ודא שהיא באמת מכסה את מה ששינית, לא רק עוברת במקרה.
- **רעננות**: הרץ את הבדיקה על הקוד הנוכחי. אל תסתמך על תוצאה מהרצה קודמת.
- אם בדיקה דורשת קונקטור שאינו מחובר — הפעל את נוהל ה-fallback (ראה
  [`connector-policy.md`](./connector-policy.md)); אל "תאמת" מהזיכרון.

### אכיפה פיזית (מותקנת ע"י `use-in-project.sh`)

> הכללים הבאים מגובים ב-hooks עם `exit 1` — לא תזכורות:
>
> - **pre-commit**: סריקת filesystem פיזית — commit נחסם אם >2 code files staged ו-0 test files בכל הפרויקט
> - **commit-msg**: commit נחסם ללא ✅❌🔄🧪 sections, או אם 🧪 section אומר "none"
> - **Write/Edit hook**: כתיבת code files נחסמת אם אין `.claude/plans/*.md`
> - **Agent hook**: spawn נחסם אם אין `.claude/tasks.json`
> - **Bash hook**: יצירת branch שני נחסמת (one-branch policy)

### בדיקת קוד קבועה (בכל קומיט, ללא תלות במשימה)

תמיד הרץ את שני אלה על ה-diff לפני קומיט:

<!-- שני הכלים הבאים טרם חוברו — להסיר את ההערה כשהחיבור יושלם:
- **CodeRabbit** — סקירת ה-diff לתפיסת באגים וכשלי אבטחה.
- **פלאגין בדיקת הקוד של קלוד** — סקירת קוד נוספת (ייחובר בהמשך).
-->

> כרגע CodeRabbit והפלאגין של קלוד עדיין לא מחוברים. עד שיחוברו — בצע סקירת קוד
> ידנית של ה-diff (קריאה ביקורתית + הרצת linter/טסטים). אם נדרשת בדיקה אוטומטית
> ואין כלי מחובר — עצור ועדכן את המשתמש.

> **אכיפה:** הרצת ה-linter והטסטים על ה-diff אינה אמורה להישען על זיכרון — הגדר אותה
> כ-pre-commit **hook אוכף** שחוסם קומיט לא תקין (ראה [`hooks-policy.md`](./hooks-policy.md)).
> הסקירה הביקורתית (CodeRabbit/ידנית) היא שכבה נוספת מעל זה, לא תחליף לה.

> **כשמפעילים reviewer אוטומטי (CodeRabbit/פלאגין):** reviewer שהתבקש למצוא פערים כמעט
> תמיד ימצא, גם כשהעבודה תקינה — וזה מוביל ל-over-engineering (שכבות אבסטרקציה מיותרות,
> קוד הגנתי, טסטים למקרים שלא יכולים לקרות). הנחה אותו לסמן **רק** פערים שפוגעים בנכונות
> או בדרישות שאופיינו; את השאר התייחס כאופציונלי. זה מתיישב עם עקרון ה-minimal-change.

### בדיקות תלויות-משימה

בנוסף, הרץ את הבדיקות הרלוונטיות לסוג המשימה:
- **לוגיקה / backend** — הרצת טסטים (unit / integration); אימות נתונים מול Supabase.
  במשימה שנוגעת ל-schema/טבלאות ב-Supabase — אמת גם ש-RLS דולק ושקיימת policy מפורשת
  (ראה [`connector-policy.md`](./connector-policy.md) › Supabase › אבטחה).
- **flows ב-web** — Playwright.
- **אפליקציית mobile** — Maestro.
- **רכיבי UI / ויזואלי** — Storybook + Chromatic.
- **פיצ'רי AI / LLM** — eval דרך Braintrust / ניטור Arize.
- **ML** — אימות על סט בדיקה (metrics: accuracy/loss וכו'); ודא שאין דליפת נתונים
  (data leakage) בין train ל-test.
- **אוטומציה / סקריפטים** — הרצה מקצה-לקצה על קלט מייצג; אימות התוצר בפועל
  (קובץ/שורה/קריאת API שנוצרו).
- **באגים בפרודקשן** — אימות מול Sentry (ראה [`debugging-policy.md`](./debugging-policy.md)).

זו רשימה פתוחה. לסוג משימה שאינו מופיע — בחר את הבדיקה שמוכיחה בפועל שהתוצר עובד,
ותעד אותה בקומיט. אם אין דרך לאמת — עצור ועדכן את המשתמש.

אם בדיקה כלשהי נכשלת — תקן לפני הקומיט, אל תתעד "עובד" בלי הוכחה.

### eval מצטבר

חבילת הבדיקות לא נשארת קבועה — היא **גדלה עם הלקחים**. כל באג משמעותי שנפתר מוסיף
טסט רגרסיה (ראה [`debugging-policy.md`](./debugging-policy.md) › `<debug_loop>` ו-
[`learning-loop.md`](./learning-loop.md)). כך ה-eval אינו פר-משימה בלבד אלא מצטבר:
ככל שהמערכת רצה, היא בודקת את עצמה מול יותר מקרי-כשל שכבר נתקלנו בהם. הרץ את חבילת
הרגרסיה כחלק מהבדיקה הקבועה לפני קומיט, כך שלקח שנלמד פעם לא נשבר בשקט בהמשך.

</pre_commit_review>

## <definition_of_done>

משימה מוצלחת = עושה את מה שאופיין, מוכח בבדיקה, ולא שברה דבר קיים. אין "כנראה
עובד" — כל תנאי נסגר בראיה. משימה גמורה רק אם **כל** התנאים מתקיימים:

- [ ] הקוד עובד ועושה את מה שנדרש (לפי קריטריוני הסיום שבאפיון)
- [ ] התוצר תואם לאפיון המקורי (ראה [`workflow.md`](./workflow.md) › `<spec_loop>`)
- [ ] הבדיקות עברו (unit / E2E לפי הרלוונטיות)
- [ ] **התוצר אומת דרך הכלי המתאים למשימה** — לא רק UI/API, אלא לפי הסוג: DB / ML /
      אוטומציה / LLM / mobile וכו' (ראה `<pre_commit_review>` › בדיקות תלויות-משימה).
      אם אין דרך לאמת — עצור ועדכן את המשתמש.
- [ ] אין רגרסיות
- [ ] בוצע ניקוי קוד מאומת (`<cleanup>`)
- [ ] בוצעה בדיקת קוד לפני קומיט (`<pre_commit_review>`)
- [ ] קומיט תקין לפי הפרוטוקול (ראה [`git-policy.md`](./git-policy.md) › `<commit_protocol>`)
- [ ] מוזג ל-main רק לאחר אישור המשתמש (ראה [`git-policy.md`](./git-policy.md))
- [ ] אם נחשף לקח משמעותי — תועד ב-`lessons-learned/` (ראה [`learning-loop.md`](./learning-loop.md))
- [ ] אם תוקן באג — נוסף טסט רגרסיה שמשחזר אותו (eval מצטבר, ראה `<pre_commit_review>`)

> ניתן לגבות את סגירת הרשימה הזו ב-Stop hook שמאמת את תנאי הסיום לפני סימון המשימה
> כגמורה (ראה [`hooks-policy.md`](./hooks-policy.md) › `<hooks>`).

</definition_of_done>
